# Как выложить сайт в интернет (для доступа друзей)

Сайт можно разместить на **Render.com** (бесплатный тариф). Данные хранятся в **GitHub** — полностью бесплатно, без карты. После деплоя всё работает по одному адресу: интерфейс, API, портреты.

---

## 1. Репозиторий на GitHub

- По желанию: в `public/` положите `img2.jpg` для фона на странице входа (если нет — тёмный фон).
1. Создайте репозиторий на [github.com](https://github.com).
2. В корне проекта `dnd-site` выполните:

```bash
cd dnd-site
git init
git add .
git commit -m "DnD site"
git branch -M main
git remote add origin https://github.com/ВАШ_ЛОГИН/ИМЯ_РЕПО.git
git push -u origin main
```

(Подставьте свой логин и имя репозитория.)

---

## 2. GitHub: токен и хранилище базы

Чтобы данные не пропадали при пересборке, база хранится в репозитории через GitHub API.

1. **Personal Access Token (PAT):** [github.com](https://github.com) → **Settings** → **Developer settings** → **Personal access tokens** → **Tokens (classic)** → **Generate new token**. Право **`repo`**. Сохраните токен — он показывается один раз.
2. Запомните формат репозитория: `логин/имя-репо` (например `ivanov/dnd-campaign`). Эти значения понадобятся в Render.

Файл `data/db.json` создаётся при первой записи. **Портреты хранятся отдельно** в `data/portraits/{id}.json` — так `db.json` остаётся меньше 1 МБ (лимит GitHub Contents API). Без этого при нескольких персонажах с портретами файл переполнялся и данные могли стираться.

**Восстановление из бэкапа** (если сохранили старый `db.json` с портретами внутри): запустите миграцию один раз — см. раздел **«Миграция бэкапа»** в конце документа.

---

## 3. Деплой на Render

1. [render.com](https://render.com) → войдите через GitHub.
2. **Dashboard** → **New** → **Web Service**.
3. Подключите репозиторий → **Connect**.
4. Настройки:

| Поле | Значение |
|------|----------|
| **Name** | например `dnd-campaign` |
| **Region** | ближайший (например Frankfurt) |
| **Root Directory** | пусто |
| **Runtime** | Node |
| **Build Command** | `npm install && npm run build && cd server && npm install` |
| **Start Command** | `cd server && node index.js` |

5. **Environment** — добавьте:

| Key | Value |
|-----|-------|
| `JWT_SECRET` | длинная случайная строка (например: `openssl rand -hex 32`) |
| `GITHUB_TOKEN` | ваш PAT из раздела 2 |
| `GITHUB_REPO` | `логин/имя-репозитория` (тот же репо или любой с правом `repo` у токена) |

6. **Create Web Service**.

После сборки появится URL вида `https://dnd-campaign-xxxx.onrender.com`.

---

## 3.1. Как обновить сайт после изменений в коде

Чтобы сайт заработал с новыми изменениями (после правок в коде):

1. Отправьте код в GitHub:
   ```bash
   git add .
   git commit -m "Описание изменений"
   git push origin main
   ```
2. Render автоматически запустит новый деплой (если репозиторий уже подключён). Либо в [dashboard.render.com](https://dashboard.render.com) откройте свой сервис → **Manual Deploy** → **Deploy latest commit**.
3. Дождитесь окончания сборки (1–3 минуты). После успешного деплоя сайт будет работать с обновлённым кодом. Данные в `data/db.json` при этом сохраняются.

## 4. Как заново развернуть проект

Если нужно развернуть с нуля (новый сервис на Render, другой репо, переустановка):

### 4.1. Репозиторий

- Код должен быть в GitHub. Если репо уже есть — просто делайте `git push` при изменениях.
- Для **нового** репо: создайте репозиторий, в папке `dnd-site` выполните `git remote set-url origin https://github.com/НОВЫЙ_ЛОГИН/НОВЫЙ_РЕПО.git`, затем `git push -u origin main`.

### 4.2. Render: новый Web Service

1. **Dashboard** → **New** → **Web Service**.
2. Подключите нужный репозиторий.
3. Настройки — как в разделе 3 (Build Command, Start Command, Root Directory пусто).
4. **Environment:** обязательно `JWT_SECRET`, `GITHUB_TOKEN`, `GITHUB_REPO`. Без `GITHUB_TOKEN` и `GITHUB_REPO` данные будут в файлах и при пересборке пропадут.
5. **Create Web Service**.

### 4.3. Render: обновление существующего сервиса

- **Деплой из кода:** после `git push` в подключённый репо Render сам пересобирает и перезапускает сервис. Данные в `data/db.json` в GitHub сохраняются.
- **Ручной пересборка:** в карточке сервиса → **Manual Deploy** → **Deploy latest commit**.
- **Смена переменных:** **Environment** → измените `GITHUB_TOKEN` или `GITHUB_REPO` → **Save Changes**. Сервис перезапустится; если `GITHUB_REPO` указывает на тот же репо с `data/db.json`, данные подхватятся.

### 4.4. Важно при смене репо

Если `GITHUB_REPO` меняете на **другой** репозиторий, в нём изначально нет `data/db.json`. При первом логине создастся пустая база. Чтобы перенести старые данные, нужно вручную добавить в новый репо файл `data/db.json` с содержимым из старого (скопировать через веб-интерфейс GitHub или `git`).

## 6. Ограничения бесплатного тарифа Render

- **«Засыпание»:** после ~15 минут без обращений следующий запрос может идти 30–60 секунд.
- **Данные:** при заданных `GITHUB_TOKEN` и `GITHUB_REPO` база и портреты в GitHub и **не пропадают** при пересборке. Без них — только файлы на диске Render, при перезапуске обнуляются.
- **Трафик:** лимиты есть; для небольшой компании обычно хватает.

---

## 6.1. Лимиты GitHub и возможные проблемы

### Сколько персонажей можно завести

- **Ограничение по количеству:** в одной папке GitHub рекомендуется не более **3000 файлов**. Портреты лежат в `data/portraits/` — по одному файлу на персонажа с портретом. Значит, **до ~3000 персонажей с портретами** упираетесь в лимит папки. Персонажи без портрета в эту папку не пишутся, их число ограничено только размером `data/db.json` (он остаётся небольшим).
- **Размер репо:** GitHub советует держать репо до 1–5 ГБ. Сотни персонажей с портретами (по ~100–300 КБ на файл) дают десятки–сотни МБ — до лимита далеко.
- **Итог:** для друзей и кампаний (десятки–сотни персонажей) лимитов GitHub хватает. Проблемы начнутся только при тысячах персонажей с портретами.

### Лимит запросов к API

- **5000 запросов в час** с вашим токеном (PAT). Одно сохранение персонажа с портретом — несколько запросов (чтение/запись `db.json` и портрета). При обычном использовании (редкие правки) лимит не достигается. Массовое создание/редактирование десятков персонажей подряд теоретически может упереться в лимит — тогда нужно подождать до следующего часа.

### Какие ещё проблемы могут возникнуть

| Проблема | Что делать |
|----------|------------|
| **Ошибка 422 при сохранении** | Часто из‑за гонки: два сохранения почти одновременно, устаревший `sha`. Повторите сохранение (кнопка «Сохранить» ещё раз). |
| **401 / 403 от GitHub** | Истёк или отозван PAT, либо у токена нет права `repo`. Создайте новый PAT с `repo` и обновите `GITHUB_TOKEN` в Render. |
| **Сайт «завис», потом пусто** | Возможен сбой GitHub или сети. Проверьте [status.github.com](https://www.githubstatus.com/). После восстановления GitHub данные в репо остаются. |
| **Медленно открываются портреты** | Каждый портрет — отдельный запрос к API. При большом числе персонажей на одной странице запросов много; можно уменьшить размер/качество загружаемых портретов. |
| **Репо стал очень большим** | При тысячах персонажей репо может тормозить. Имеет смысл вынести данные в отдельный репо только под базу и портреты, либо подумать о другой хранилище (БД и т.п.). |

Для типичного использования (одна–несколько кампаний, до сотни персонажей) текущей схемы с GitHub достаточно.

1. Войти как `admin` / `6852`.
2. В админке создать нового пользователя с надёжным паролем.
3. При желании завести отдельного админа и удалить `admin` вручную в `data/db.json` (в GitHub) или не использовать.

## 8. Как это устроено

- **Build:** `npm run build` (Vite → `dist/`), `cd server && npm install`.
- **Start:** `node server/index.js`. При `GITHUB_TOKEN` и `GITHUB_REPO` — чтение/запись в `data/db.json` и портреты в `data/portraits/{id}.json` в репо; иначе — `server/data/db.json` и `server/uploads/` (на Render при пересборке теряются).
- Фронт отдаётся из `dist/`; SPA-маршруты — `index.html`; `/api` и `/uploads` — backend.

Локально: `npm run dev` (фронт) и `node server/index.js` (бэк). С GitHub — нужен доступ к api.github.com.

## 9. Миграция бэкапа (восстановление после поломки)

Если у вас есть сохранённый файл `db.json` **до поломки** (с портретами в base64 внутри), его нельзя просто загрузить в репо — он больше 1 МБ и GitHub отклонит. Нужно один раз вынести портреты в отдельные файлы и загрузить облегчённый `db.json`.

1. В корне проекта (или в папке `server`) выполните:
   ```bash
   cd dnd-site/server
   set GITHUB_TOKEN=ваш_токен
   set GITHUB_REPO=логин/имя-репо
   node scripts/migrate-db-portraits.js "C:\Users\biv\Downloads\db (2).json"
   ```
   (В PowerShell: `$env:GITHUB_TOKEN="..."; $env:GITHUB_REPO="..."; node scripts/migrate-db-portraits.js "путь\к\файлу.json"`.)

2. Скрипт создаст в репо файлы `data/portraits/{id}.json` для каждого персонажа с портретом и загрузит облегчённый `data/db.json` (без base64 в персонажах).

3. После этого задеплойте сайт (или просто перезапустите сервис). Данные и портреты подхватятся из репо.
